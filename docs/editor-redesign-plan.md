# 后台文章编辑器改造方案

> 状态：待确认方案。本文只定义改造方向、范围和阶段，不改变 Markdown 输入 / TipTap JSON 保存的既有契约。

## 1. 背景与结论

当前后台编辑器已经具备 TipTap 编辑、Markdown 粘贴导入、正文样式复用和部分 rich block 可视化编辑能力，但整体体验仍然不像文档编辑工具，更像“后台表单 + 一个很重的富文本框”。这会导致写作时视觉负担高、插入组件流程割裂、文章元信息抢占主编辑注意力，最终不适合作为长期写作工作台。

改造目标不是把后台做成正式文章详情页预览，也不是继续给现有编辑框补按钮。目标是做一个接近飞书 / Lark 文档编辑器的后台写作工作台：标题和正文是主舞台，工具在需要时出现，文章设置和组件属性在侧边完成，Markdown 和高级能力保留但不干扰主写作流。

## 2. 现状问题诊断

### 2.1 信息架构像后台表单，不像写作工作台

当前 `app/admin/posts/new/page.tsx` 和 `app/admin/posts/[id]/page.tsx` 都以多张 `Card` 组织标题、slug、摘要、SEO、封面、内容、发布设置和标签。结果是：

- 标题不是文档的一部分，而是普通表单字段。
- 正文被放进“内容”卡片里，像一个嵌入控件，而不是页面核心。
- 新建页和编辑页结构、宽度、字段顺序不一致，后续维护和体验统一都会变难。
- SEO、封面、标签等低频设置在主流程里占据太多视觉权重。

### 2.2 编辑器外壳过重

`features/blog/editor/tiptap-editor.module.css` 里 `editorCanvas` 使用边框、圆角、背景和较大的容器感，`ProseMirror` 又居中在内部。这让编辑区域看起来像一个“文章预览框”或“表单控件”，而不是自然铺开的文档画布。

需要保留正文排版能力，但后台外壳不能照搬正式文章阅读页。后台应该只复用排版 token、基础内容组件和节点渲染能力，不复用文章页的阅读框架感。

### 2.3 工具条过密，暴露时机错误

`features/blog/editor/menu-bar.tsx` 当前常驻展示大量格式按钮和 rich block 插入按钮。问题不在于按钮是否齐全，而在于它们同时出现：

- 常驻 toolbar 在写作前就占据注意力。
- rich block 插入入口全部摊开，用户需要记图标含义。
- 文字格式、块级插入、媒体上传、撤销重做混在同一层。
- 已经有 `SelectionBubbleMenu`，但常驻 toolbar 仍然太重，弱化了选区浮层的价值。

### 2.4 插入和编辑 rich block 的方式割裂

当前很多插入动作依赖 `window.prompt`，例如 callout、button、gallery、embed 等。这是原型式交互，不适合正式编辑：

- 用户插入前看不到结构，也不能撤回单个字段。
- 多字段组件需要连续弹窗，流程脆弱。
- 组件配置和文档上下文分离。
- 可视化编辑面板直接展开在 rich block 内部，复杂组件会把正文节奏撑乱。

### 2.5 Markdown 能力位置不对

Markdown 粘贴导入和 shortcode guide 是必要能力，但当前 `MarkdownShortcodeGuide` 直接出现在编辑器内部。它更像高级帮助信息，不应该长期占据正文上方。Markdown 应该是导入入口、粘贴能力和高级帮助，而不是主写作界面的常驻模块。

## 3. 设计原则

1. **写作优先**：标题和正文是第一层界面，其他元信息收进侧边或二级入口。
2. **工具按需出现**：选中文本时显示文字格式浮层，hover 空行或块时显示块级操作，输入 `/` 召出插入菜单。
3. **正文不是预览框，但内容必须一致**：后台不能套正式文章页的外壳、装饰和阅读页结构；但正文内部的段落、标题、表格、代码块、媒体和 rich block 渲染必须与正式发布页一致。
4. **组件可视化编辑**：自定义 rich block 必须能在后台可视化创建和编辑，不能要求用户改源码、JSON 或 shortcode。
5. **存储契约不变**：Markdown 继续作为输入 / 导入格式；TipTap JSON 继续作为保存格式；API 和数据库内容结构不因 UI 改造而变化。
6. **共用渲染能力，不共用工作台外壳**：正文页和后台编辑器共享节点样式、内容组件、基础 CSS token；后台单独拥有编辑态工具、属性面板和交互状态。
7. **禁止双实现漂移**：不能为后台编辑器另写一套“看起来差不多”的文章组件。编辑态只能在正式渲染组件外叠加选择框、hover 控件、插入按钮和属性入口。

## 4. 目标体验

### 4.1 页面结构

后台文章编辑页改为统一的 `PostEditorWorkspace`：

- 顶部轻量栏：返回、保存状态、草稿保存、发布 / 更新、更多操作。
- 主文档区：标题输入、摘要轻提示、正文编辑器。
- 右侧面板：发布设置、标签、封面、SEO、语言、统计、当前选中 rich block 属性。
- 高级入口：Markdown 导入、shortcode 帮助、JSON 调试只放在更多菜单或右侧面板的高级分组。

新建页和编辑页应复用同一套工作台组件，只在数据加载、删除按钮、提交方法上有差异。

### 4.2 主文档区

主文档区应该像文档编辑器：

- 标题是大号无边框输入，位于正文上方。
- slug 自动生成放到右侧“基础信息”或标题旁的小状态里，不阻断写作。
- 正文区域不再被一个明显卡片框住。
- placeholder 应提示“输入 / 插入内容块，或直接粘贴 Markdown”。
- 字数、保存状态可以轻量显示在顶部栏或右下角，不需要占据正文底部边框。
- 正文内容本身必须使用正式发布页同源的排版和组件渲染，编辑器外壳只负责工作台布局和编辑控件。

### 4.3 工具交互

工具分三层：

- **选区浮层**：只处理文字格式，例如加粗、斜体、链接、标题、引用、行内代码。
- **块级控制**：hover 当前块时显示拖拽手柄、插入 `+`、更多菜单。先实现 `+` 和更多菜单，拖拽排序可作为后续阶段。
- **Slash 菜单**：输入 `/` 或点击空行 `+` 时打开插入菜单，按“基础块、媒体、文章组件、结构组件”分组。

常驻 toolbar 不再作为主入口。可以保留一个极轻的顶部 command button，但不应把所有按钮展开。

### 4.4 参考飞书截图的目标形态

用户给出的飞书 / Lark 截图可以作为后台编辑器的视觉和交互参考。这里参考的是“文档编辑体验”，不是完整复刻飞书产品壳。

可借鉴的部分：

- **三栏感知**：左侧为文章大纲和当前文档导航，中间为文档画布，右侧为发布设置、属性面板或暂时收起。
- **暗色文档工作台**：整体背景低对比，正文区域没有卡片边框，标题、正文、表格和块级控件自然处在同一画布中。
- **文档标题即主输入**：标题以大号文本出现在正文顶部，而不是“基础信息”卡片里的普通输入框。
- **元信息轻量化**：作者、更新时间、保存状态、发布状态可以放在标题下方或顶部栏，视觉权重低于正文。
- **块级 `+` 插入入口**：当前块左侧或空行左侧出现 `+`，点击后打开插入菜单。
- **插入菜单分组**：菜单按基础、常用、媒体、表格、文章组件、模板等分组，支持滚动，不把所有能力铺在顶部。
- **选区菜单独立**：选中文字后出现小型浮层，只承载文本格式、评论、链接等上下文动作。
- **块选中态克制**：选中或 hover 块时使用细边框、浅色背景或左侧控件提示，不用大面积表单样式打断正文。
- **表格和代码等原生块直接可编辑**：基础块应在正文里直接编辑，只有复杂文章组件才进入右侧属性面板或弹层。

不照搬的部分：

- 不复制飞书完整的知识库、云盘、组织空间、分享协作和头像体系。
- 不把博客后台左侧做成复杂文档树；第一阶段只需要文章返回入口、当前文章大纲和必要导航。
- 不实现多人协同光标、评论流、权限分享和在线状态，除非后续作为独立需求设计。
- 不让飞书样式覆盖前台文章页；后台工作台样式与前台正文样式共享 token，但保持编辑态边界。

针对这两张截图，第一版可验收目标是：

- 打开 `/admin/posts/new` 后，第一眼看到的是文档标题和正文画布，而不是多张表单卡片。
- 正文左侧有大纲或块级导航，标题层级能被看见。
- 默认状态没有一整排常驻富文本按钮。
- 在空行或块左侧能看到 `+` 入口，点击后出现类似飞书的分组插入菜单。
- 选中文字后出现独立格式浮层。
- 插入 rich block 后看到真实块渲染，而不是先看到一组表单字段。

### 4.5 发布渲染一致性

这是本次改造的硬性要求：后台编辑器里看到的正文内容和正式发布后的正文内容必须一致。这里的“一致”指内容块本身的视觉结构、字体层级、间距、颜色 token、响应式行为和组件状态一致；不要求后台工作台外壳、顶部栏、左侧大纲、右侧属性面板与前台文章页一致。

实现规则：

- 正式发布页和后台编辑器应共享同一套正文样式入口，例如 `app/styles/article.css`、`app/styles/article-components.css`、`app/styles/article-skin.css` 中的内容 token 和组件样式。
- rich block 的展示部分应抽成可复用 renderer，前台文章页和后台 NodeView 共同调用；后台 NodeView 只额外包一层编辑态容器和控制按钮。
- 原生内容块也要一致：标题、段落、列表、引用、代码块、表格、图片、任务列表、数学公式在后台和发布页使用同一套 class/token。
- 编辑态控件必须与内容渲染分离：`+`、拖拽手柄、选中框、编辑按钮、属性面板入口不能改变发布态组件 DOM 的核心结构和样式语义。
- 不允许后台为了“方便编辑”把 rich block 降级成表单卡片常态显示。默认展示必须是真实发布块，编辑入口按 hover、选中或右侧属性面板出现。
- 发布前预览不应成为唯一校验手段；主编辑器本身就要接近最终输出。

验收标准：

- 同一份 TipTap JSON 在后台编辑器和正式文章页渲染时，正文块视觉应只差编辑控件和工作台背景。
- 对每类 rich block 至少检查一次后台编辑态与发布页展示：`callout`、`button`、`tabs`、`accordion`、`gallery`、`slider`、`embed`、`flow`、`cards`、`diagram`、`timeline`、`rich showcase`、`audio`。
- 桌面和移动端都要检查表格、代码块、图片和复杂 rich block，避免后台看起来正常但发布页换行或宽度不同。
- 如果某个组件必须在编辑态有额外辅助 UI，应确保辅助 UI 可明确隐藏，且不影响发布态渲染。

## 5. Rich block 编辑策略

rich block 的编辑方式按复杂度分层，而不是全部在正文里展开表单。

### 5.1 内联编辑

适合简单、短内容、与正文节奏高度相关的块：

- `callout`：tone、title、text。
- `button`：label、href、variant。
- `audio`：title、caption、src 的轻量编辑。

这些块可以在节点内提供小型编辑态，但默认只显示内容本身和一个 hover 编辑按钮。

### 5.2 右侧属性面板

适合多字段、多列表或会撑乱正文的块：

- `tabs`
- `accordion`
- `gallery`
- `slider`
- `cards`
- `diagram`
- `timeline`
- `rich showcase`

点击块后，正文中只显示选中态和真实渲染，右侧面板切换到该块的属性编辑。列表项增删、排序、图片 URL、alt、标题、说明都在面板中完成。

这些块的正文区域仍必须显示正式发布效果。右侧属性面板只负责修改 attrs，不替代正文中的真实渲染。

### 5.3 专用弹层 / 抽屉

适合创建时需要较多初始配置或外部资源选择的块：

- `embed`：YouTube / video URL 解析、poster、title。
- 图片上传 / 批量导入。
- Markdown 批量导入。

这些不应该用 `window.prompt`，应使用项目现有 dialog / drawer 或后续新增的轻量面板组件。

### 5.4 组件注册表

为 rich block 建立一个本地 registry，统一描述：

- TipTap node type。
- 菜单标题、图标、分组、说明。
- 默认 attrs。
- 创建入口。
- 属性面板组件。
- 发布态 renderer。
- 编辑态 NodeView wrapper。
- 是否支持内联编辑。

这样可以替代 `menu-bar.tsx` 里一长串硬编码插入函数，也方便后续新增组件。

## 6. Markdown 与高级入口

Markdown 继续保留三种能力：

1. 粘贴 Markdown 自动转换为 TipTap 内容。
2. 从“导入 Markdown”抽屉批量导入。
3. 高级帮助中展示 shortcode 语法。

但默认编辑界面不显示 shortcode guide。用户需要时从顶部更多菜单或右侧高级面板打开。

导入后的内容仍写入 TipTap JSON；保存时仍保存 JSON，不新增 Markdown 存储字段。

## 7. 分阶段实施计划

### 阶段 0：方案确认与基线清理

目标：确定改造边界，避免继续在旧 toolbar 上堆功能。

- 保留当前已实现的 TipTap extension、Markdown paste、rich block NodeView 能力。
- 不再继续扩展常驻 toolbar。
- 确认新建页和编辑页要合并到同一工作台组件。
- 验证当前分支可 lint/typecheck，作为回滚基线。

验收：

- 本文档确认后再开始代码改造。
- 不改变现有 API、数据库、TipTap JSON 内容结构。

### 阶段 1：统一文章编辑工作台

目标：先解决“后台表单感”和新建 / 编辑不一致的问题。

建议新增或重组：

- `features/admin/posts/post-editor-workspace.tsx`
- `features/admin/posts/post-editor-sidebar.tsx`
- `features/admin/posts/post-editor-topbar.tsx`
- `features/admin/posts/post-editor-shell.module.css` 或沿用当前 styling 约定

主要改动：

- 新建页和编辑页共用 `PostEditorWorkspace`。
- 标题、摘要、正文进入主文档流。
- slug、封面、SEO、标签、发布设置、语言、统计进入右侧面板。
- 去掉主编辑区域外层大卡片感。

验收：

- `/admin/posts/new` 和 `/admin/posts/[id]` 视觉结构一致。
- 保存草稿、发布、更新、删除逻辑保持原行为。
- 移动端右侧面板降级为抽屉或下方设置区，不产生横向滚动。

### 阶段 2：重做编辑器工具层

目标：把常驻密集 toolbar 改为 Lark 式按需工具。

主要改动：

- 缩小或移除 `MenuBar` 常驻按钮。
- 保留并强化 `SelectionBubbleMenu`。
- 新增块级插入入口：空行 `+`、块 hover `+`。
- 新增 slash command 菜单：基础块、媒体、文章组件、结构组件分组。
- 撤销 / 重做移动到顶部栏或快捷键优先。

验收：

- 默认写作状态下没有横向展开的大量按钮。
- 选中文本后只出现文字相关浮层。
- 点击 `+` 或输入 `/` 可以插入基础块和 rich block。
- 插入流程不再使用 `window.prompt`。

### 阶段 3：rich block 渲染复用与属性面板

目标：让复杂组件真正可视化编辑，同时保证后台正文和正式发布页使用同源渲染，不污染正文节奏。

主要改动：

- 建立 rich block registry。
- 抽出每个 rich block 的发布态 renderer，让前台文章页和后台 NodeView 共用。
- 为每个 rich block 定义属性编辑组件。
- 简单块保留内联轻编辑，复杂块改用右侧属性面板。
- 当前选中块与右侧属性面板联动。
- NodeView 只做编辑态包装：选中态、hover 编辑按钮、属性面板联动，不另写一套展示 UI。

验收：

- `callout`、`button` 可快速内联编辑。
- `tabs`、`accordion`、`gallery`、`slider`、`cards`、`diagram`、`timeline`、`rich showcase` 可在右侧面板编辑。
- `embed`、图片集合、Markdown 导入使用抽屉或弹层。
- 所有 rich block 修改仍写回原 TipTap JSON attrs。
- 每个 rich block 的后台展示和发布页展示只差编辑态控件，不差组件样式和内容结构。

### 阶段 4：Markdown 导入与高级能力收纳

目标：保留高级能力，但不让它压住主写作界面。

主要改动：

- `MarkdownShortcodeGuide` 从正文顶部移到高级帮助入口。
- 新增 Markdown 导入抽屉：粘贴 Markdown、预检查、导入到当前光标位置或替换正文。
- JSON 调试如有需要，只放在开发 / 高级区域，不默认暴露。

验收：

- 粘贴 Markdown 仍可自动转换。
- 用户可通过明确入口导入 Markdown。
- 默认编辑页不出现 shortcode 大块说明。

### 阶段 5：视觉、响应式与验证

目标：把工作台打磨到可长期使用。

检查项：

- 桌面端：主编辑区、右侧面板、顶部栏的固定和滚动关系稳定。
- 移动端：标题、正文、设置入口可用，无横向滚动。
- 键盘：Tab、Esc、Enter、`/`、保存快捷键不冲突。
- 状态：loading、saving、saved、error、validation error 有明确反馈。
- 无障碍：按钮有 label，浮层可键盘关闭，焦点不丢失。

建议验证：

- `pnpm run lint`
- `pnpm exec tsc --noEmit`
- 至少手动检查 `/admin/posts/new` 和一个 `/admin/posts/[id]`
- 用同一篇包含标题、表格、代码块、图片和所有 rich block 的测试文章，对比后台编辑态和正式文章页显示。
- UI 改完后用截图检查桌面和移动端主路径

## 8. 风险与回滚点

### 风险

- TipTap NodeView 和右侧属性面板联动可能引入选中态同步问题。
- Slash command 菜单如果一次做太多，容易变成新的复杂 toolbar。
- 正文样式复用要控制边界，避免后台再次变成文章预览页。
- 新建页和编辑页合并过程中，可能遗漏删除、加载、成功跳转等编辑页特有流程。

### 回滚点

- 阶段 1 可独立回滚到旧表单页面，不影响 TipTap 内容。
- 阶段 2 可独立回滚工具层，不影响 rich block 数据。
- 阶段 3 只修改编辑 UI 和 attrs 写入方式，不改变节点类型和保存结构。
- Markdown 导入能力在阶段 4 独立迁移，失败时可恢复旧 guide 入口。

## 9. 明确不做

- 不改变文章内容的数据库字段和 API 契约。
- 不把后台编辑器做成正式文章详情页预览。
- 不要求用户编辑 JSON、源码或 shortcode 才能配置 rich block。
- 不引入新的大型编辑器框架替换 TipTap。
- 不在本轮顺手重构博客正文页、前台文章页或无关后台页面。

## 10. 下一步决策点

开始实现前需要确认：

1. 右侧面板是否默认常驻，还是可折叠。
2. 移动端设置面板使用抽屉还是折叠区。
3. rich block 复杂编辑优先级：先做 `callout/button/gallery/embed`，还是一次覆盖全部现有块。
4. 是否需要第一阶段就做自动保存；如果要做，应单独设计保存节流、冲突处理和错误恢复，不混入纯 UI 改造。
